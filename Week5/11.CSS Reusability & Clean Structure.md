## 🧠 2. Class Naming (VERY IMPORTANT)

---

### ❌ Bad Naming

```css
.redText {}
.bigBox {}
.div1 {}
```

👉 Not scalable ❌

---

### ✅ Good Naming (Meaningful)

```css
.navbar {}
.card {}
.btn-primary {}
```

👉 Based on **purpose**, not appearance

---

## 🔥 3. BEM Naming (Pro Level)

👉 **Block Element Modifier**

```css
.card {}
.card__title {}
.card--highlight {}
```

---

### 🧱 Breakdown:

* Block → component (`card`)
* Element → inside (`card__title`)
* Modifier → variation (`card--active`)

---

## 🧠 4. Utility Classes (FAST DEVELOPMENT)

👉 Small reusable classes

---

### Example:

```css
.mt-20 { margin-top: 20px; }
.text-center { text-align: center; }
.flex { display: flex; }
```

---

### Usage:

```html
<div class="flex text-center mt-20">
  Hello
</div>
```

---

👉 This is how frameworks like Tailwind CSS work 🔥

---

## 🧠 5. Avoid Inline CSS (VERY IMPORTANT)

---

### ❌ Bad

```html
<div style="color: red; font-size: 20px;">
```

---

### ✅ Good

```css
.text-red {
  color: red;
}
```

```html
<div class="text-red">
```

---

### 🚨 Why avoid inline?

* Not reusable
* Hard to maintain
* Breaks clean structure

---

## 🧠 6. Component-Based Thinking

👉 Think in **components**, not pages

---

### Example:

* Navbar
* Card
* Button
* Footer

Each should have its **own styles**

---

```css
.card {
  padding: 20px;
  border-radius: 10px;
}
```

---

## 🧠 7. Folder Structure (IMPORTANT)

```id="folder1"
css/
 ├── style.css
 ├── components/
 │    ├── navbar.css
 │    ├── card.css
 │    └── button.css
```

---

👉 Clean separation = scalable projects
